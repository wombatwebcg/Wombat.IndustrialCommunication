# ConnectionPool 探活与误判掉线修复计划

## 1. 文档目标

本文档用于分析 `Wombat.IndustrialCommunication.ConnectionPool` 当前“无故断开、重连失败、设备实际在线”的问题，并给出一套稳妥的修复计划。

本次目标只有三个：

- 明确当前误判链路
- 确定探活与掉线判定的修复原则
- 给出最小但稳妥的落地步骤

本文档描述的是实施计划，不代表代码已经完成修改。

## 2. 问题现象

第三方调用时，经常出现以下现象：

- 连接池把连接判定为掉线
- 连接池主动断开并尝试重连
- 重连后仍无法恢复
- 实际 PLC / 设备并未掉线

这说明当前问题不在设备端，而在连接池的健康检查、故障判定和状态机设计。

## 3. 当前实现分析

### 3.1 当前探活路径

当前后台维护会周期性执行健康检查：

- `ConnectionPoolMaintenanceService.RunAsync`
- `PooledResourceEntry.RunHealthCheckAsync`
- `PooledResourceEntry.RunHealthCheckCoreAsync`

对已建立会话的连接，当前逻辑会：

1. 调用 `Connection.ProbeAsync`
2. 一旦探活失败，将条目标记为 `Faulted`
3. 达到连续失败阈值后，调用 `InvalidateCore`
4. 条目进入不可恢复的“判死”状态

### 3.2 当前探活的实际问题

对 `ModbusTcp` / `SiemensS7` 来说，`ProbeCoreAsync()` 当前不是轻量链路检查，而是直接执行一次真实 `ReadAsync(...)`。

这会把下面几类情况误判成掉线：

- 目标探活地址本身不稳定
- PLC 瞬时繁忙，响应变慢
- 一次读超时
- 协议脏响应或报文不同步
- 读取失败，但 TCP 会话其实仍在

换句话说，当前实现把“业务读失败”和“连接断开”混成了一件事。

### 3.3 当前状态机问题

当前实现里，存在一个“后续不再尝试恢复”的终态：

- `AcquireAsync` 遇到终态直接拒绝
- `EnsureConnectedCoreAsync` 遇到终态直接拒绝
- `TryRecoverCoreAsync` 遇到终态直接拒绝
- `ForceReconnectAsync` 最终仍走 `TryRecoverAsync`

结果就是：

- 一旦健康检查把条目打进这个终态
- 条目后续基本无法被正常恢复
- 外部看到的就是“设备在线，但连接已经彻底死掉”

## 4. 根因结论

根因不是单点，而是三件事叠加：

1. 后台探活使用真实业务读作为健康检查。
2. 单次或少量读取失败被过度解释为连接断开。
3. 健康检查失败会把条目推进不可恢复终态。

因此当前最需要修的不是“重连速度”，而是“掉线判定模型”。

## 5. 修复原则

本次按业界更稳的做法处理，原则如下。

### 5.1 读取失败不等于掉线

这是第一原则。

以下情况只能记为“操作失败”或“可疑连接”，不能直接判定掉线：

- 单次读失败
- 单次写失败
- 单次超时
- 单次协议解析失败
- 单次探活地址读取失败

只有明确的连接层证据，才允许判定为掉线，例如：

- socket 已关闭
- 远端主动断开
- 建连失败
- 连续多次主动恢复均失败

### 5.2 先轻量检查，后按需恢复

探活动作要尽量轻，不要默认发真实业务读请求。

默认顺序建议收敛为两步：

1. 轻量可用性检查
2. 失败后按现有恢复链路重连

也就是说：

- 默认探活只检查底层资源是否仍可用
- 不默认读取 `ProbeAddress`
- 探活失败后进入恢复尝试，不直接判死
- `ForceReconnect` 仍作为人工兜底手段

### 5.3 连接状态不要再加层

本次不建议先引入新的 `Degraded` 一类状态。

当前实现已经有：

- `Ready`
- `Leased`
- `Faulted`
- `Disposed`

本次只需要收紧状态语义：

- 后台维护失败时最多把条目标记为 `Faulted`
- 不应因为探活失败直接进入不可恢复终态
- 删除“后续永不再恢复”的生命周期状态

### 5.4 设备不能被钉死

这是第二条硬规则。

无论探活结果是失败，还是恢复过程失败，连接池都不能把设备条目推进到“后续永不再尝试恢复”的状态。

要求如下：

- 后台健康检查失败后，条目仍然保留在池中
- 条目仍然允许后续自动恢复
- 条目仍然允许 `ForceReconnect`
- 条目仍然允许后续业务调用触发恢复

也就是说：

- 可以判定“当前不可用”
- 但不能判定“永久不可恢复”

### 5.5 后台维护要主动恢复

本次不仅要避免误杀，还要明确恢复责任。

规则应改为：

- 探活失败后，连接池要主动尝试恢复
- 恢复失败后，进入下一轮维护时继续尝试恢复
- 恢复过程中使用退避，避免死循环狂连
- 只要条目未被人工移除或强制关闭，就一直保留自动恢复能力

### 5.6 后台维护不应制造业务扰动

后台维护的职责是观测和修复，不是制造额外故障。

因此需要避免：

- 周期性使用真实业务地址做探活
- 在短时间内频繁断开重连
- 因维护线程误判而打断正常长连接

## 6. 推荐目标模型

### 6.1 健康检查改为最小模型

建议把健康检查收敛成一条默认主路径。

默认做轻量可用性检查：

- 检查资源是否已释放
- 检查底层连接对象是否还可用
- 检查是否已出现显式断开信号

这里不要求百分百检测半开连接，但要求“不误杀”。

宁可保守一些，也不要把活连接打死。

可选增强才做协议级探活：

协议级探活保留为可选增强，而不是默认掉线判定依据。

规则应改为：

- 可以执行探活
- 仅在明确启用 `ProbeAddress` 时执行
- 失败时只作为辅助信号
- 不直接 `Disconnect`
- 不直接 `Invalidate`

## 7. 建议的状态机修正

### 7.1 删除不可恢复终态

建议直接删除这个生命周期状态。

后台健康检查、人工失效、强制关闭之后，都只进入“当前不可用但可恢复”的故障态。

### 7.2 后台失败只进入可恢复状态

后台健康检查失败后的目标状态应是：

- `Faulted`

并且应满足一点：

- 后续 `AcquireAsync`
- `EnsureConnectedCoreAsync`
- `TryRecoverCoreAsync`
- `ForceReconnectAsync`

都可以继续恢复该条目。

### 7.3 恢复失败后仍保留自动重连资格

后台恢复失败不能把条目钉死。

修复后应满足：

- 本轮恢复失败，只记录失败次数和最后失败时间
- 下一轮健康检查或后台维护到来时，仍继续尝试恢复
- 业务线程调用 `AcquireAsync` / `ExecuteAsync` 时，也允许再次触发恢复

建议语义：

- `Faulted` = 当前不可用，但会继续自动恢复
- `Disconnected` = 当前未建立连接，但允许继续自动恢复
- 不再存在“人工或强制终止后永不自动恢复”的生命周期状态

### 7.4 `ForceReconnect` 必须可救活条目

`ForceReconnect` 的职责就是“无条件尝试重建连接”。

因此修复后必须保证：

- 即使条目曾被后台打成故障态
- `ForceReconnect` 仍能执行 `DisconnectOrShutdown + EnsureAvailableAsync`
- 不依赖当前状态是否为终态

否则这个 API 名义上存在，实际上无法兜底。

## 8. 具体实施计划

### 阶段 1：先修状态机，停止不可恢复误杀

目标：

- 后台健康检查不再把探活失败推进到不可恢复终态
- `ForceReconnect` 保证能恢复故障条目
- 恢复失败后条目仍保留自动重连资格

建议改动：

1. 修改 `PooledResourceEntry.RunHealthCheckCoreAsync`
2. 去掉“健康检查失败 -> `InvalidateCore`”这条默认路径
3. 保留失败计数，但仅标记为 `Faulted`
4. 修改 `ForceReconnectAsync` / `TryRecoverCoreAsync`，允许恢复后台故障态条目
5. 后台恢复失败后不转终态，只更新失败信息并等待下轮继续恢复

这一阶段优先级最高，收益最大，改动也最小。

### 阶段 2：收紧探活动作，默认不再用业务读判掉线

目标：

- 探活从“真实业务读”改为“轻量可用性检查”

建议改动：

1. `ProbeAsync` 默认只做底层可用性检查
2. 不再默认读取 `ProbeAddress`
3. 探活失败后进入恢复尝试，不直接判死
4. 恢复失败后保留条目，等待下一轮继续恢复

### 阶段 3：把 `ProbeAddress` 降级为可选增强项

目标：

- `ProbeAddress` 不再承担“默认掉线判定”

建议改动：

1. 保留 `ProbeAddress` 配置能力
2. 但仅在明确启用协议级探活时使用
3. 文档中明确说明：
   - `ProbeAddress` 是增强观测
   - 不是强掉线判定依据
   - 不应使用高风险业务地址

### 阶段 4：按需补齐事件和诊断信息

目标：

- 让第三方能区分“读失败”和“连接恢复”

建议补充：

- 健康检查失败事件
- 恢复尝试事件
- 最近失败时间
- 最近恢复时间

这样第三方日志才能看清：

- 是业务请求失败
- 还是连接真的断了
- 还是后台维护在误操作

## 9. 默认策略建议

为了稳，建议默认值做保守化调整。

### 9.1 默认健康检查策略

- 默认启用轻量可用性检查
- 默认不把协议读探活作为掉线判定依据
- 默认空闲时也不立即断开重连

### 9.2 默认恢复策略

- 单次失败不重连
- 连续多次失败后进入恢复尝试
- 一旦进入恢复路径，后台维护持续尝试重连
- 恢复失败后保持可重试，不进入后台不可恢复终态
- 使用退避控制重试频率，而不是停止恢复

### 9.3 默认文档口径

文档需要明确写清：

- “读取失败”不等于“连接掉线”
- “探活失败”不等于“设备离线”
- `ForceReconnect` 是人工兜底恢复
- `Invalidate` 是人工明确判死

## 10. 测试计划

本次修复至少补以下测试。

### 10.1 状态机测试

- 健康检查失败后条目不能进入不可恢复死态
- `ForceReconnect` 能恢复后台故障态条目
- 人工 `Invalidate` 后仍保持终态语义
- 恢复失败后下一轮维护仍会继续尝试重连

### 10.2 探活测试

- 单次 `ProbeAsync` 失败不触发断连
- 连续多次 `ProbeAsync` 失败触发主动恢复尝试，但不直接判死
- 默认 `ProbeAsync` 不读取业务地址
- 恢复失败后后续健康检查仍继续尝试恢复

### 10.3 回归测试

- 长连接正常业务不应被后台维护打断
- 低频业务不应被空闲探活误杀
- 第三方在 `ForceReconnect` 后可重新使用连接

## 11. 推荐实施顺序

建议按下面顺序做，避免一次改太大：

1. 先删除不可恢复终态，并保证失败后持续可恢复
2. 再收紧掉线判定条件
3. 再把主动探活改成轻量检查优先
4. 最后补诊断和文档

这个顺序最稳，因为第一步就能先止血。

## 12. 一句话结论

这次问题的核心不是“不会重连”，而是“掉线判定太激进，而且判死后不可恢复”。

最稳的修法不是继续加强读探活，而是把规则改成：

- 读取失败不等于掉线
- 后台探活只做辅助观测
- 删除“后续永不再尝试恢复”的生命周期状态
- 探活失败后连接池主动尝试恢复，而且失败后继续尝试
- `ForceReconnect` 必须始终能兜底恢复
