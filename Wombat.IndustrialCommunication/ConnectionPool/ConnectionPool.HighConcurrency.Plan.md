# ConnectionPool 高并发控制面改造计划

## 目标

在“几百个离线连接同时快速 `stop/restart/recover`”场景下：

- 在线设备读写不被离线恢复风暴明显拖慢
- 控制面操作不因池级锁、同步事件、全池轮询而明显抖动
- 改造分两阶段推进，先降抖动，再补吞吐

## 现状结论

- 业务读写按连接条目隔离，不是全池单线程
- 控制面存在池级大锁内同步阻塞，批量操作会抖
- 故障恢复主要依赖后台扫描，离线量大时恢复吞吐受限
- 事件同步分发，慢订阅者会拖慢主流程
- 底层部分探测路径包含同步 `Ping`，离线多时性价比差

## 分阶段策略

### 第一阶段：先止血，降低控制面抖动

#### 目标

- 去掉最明显的全局阻塞点
- 让批量 `stop/restart/recover` 不再把主调用线程拖住
- 不大改公开接口，尽量小步落地

#### 改动项

1. 缩小 `_poolLock` 持锁范围

- `ResourcePool` 中 `_poolLock` 只负责：
  - 查找条目
  - 标记待移除
  - 从 `_entries` 摘除
- 禁止在 `_poolLock` 内执行：
  - `PrepareForRemovalAsync(...).GetAwaiter().GetResult()`
  - `DisposeAsync(...).GetAwaiter().GetResult()`
  - 其他慢 IO 或异步等待

2. 控制面事件改异步出队

- 为连接池事件增加内部事件队列
- 主流程只入队，不同步广播订阅者
- 后台单独 dispatcher 负责逐个分发
- 保持现有事件模型不变，先不改 public event 签名

3. 删除或绕过同步 `Ping`

- 移除 `BasePooledDeviceClientConnection` 中同步 `Ping.Send(host, 1000)` 依赖
- 连接可用性以协议层 `ConnectAsync/ProbeAsync` 为准
- 如果必须保留，至少改成默认关闭

4. 明确修正半开连接判定

- 考虑“网线拔掉但 socket 仍显示 `Connected`”场景
- `Connected` 只作为弱信号，不再视为连接真实可用的充分条件
- 空闲连接依赖 `ProbeAsync` 做协议级探活
- 业务执行中一旦出现超时、socket 异常、连接关闭等可恢复错误，立即标记 `Faulted`
- 为后续恢复队列提供更及时、准确的故障摘除信号

5. 调整租约过期策略，避免长执行被误回收

- 当前租约过期时间在获取时一次性写死，长时间 IO 可能被后台误判为过期
- 第一阶段不做复杂自动续约
- 改为：执行中不回收 lease，执行结束后恢复普通过期规则
- 过期扫描时如果条目存在活跃执行，则跳过 lease 回收
- 目标是先避免“长批量读写过程中 lease 被后台释放”

6. 增加批量控制操作的内部调度入口

- 先不急着暴露完整新 API
- 先引入内部控制任务入口，供后续批量 `force close/reconnect` 复用
- 目标是让第二阶段能直接接 worker/queue

#### 第一阶段不做

- 不重写整个状态机
- 不改业务读写接口
- 不做复杂优先级调度
- 不追求离线恢复吞吐最大化，只先解决“抖”

#### 第一阶段验收

- 批量 `Unregister/Dispose/ForceClose/ForceReconnect` 时，不再出现明显的池级串行卡顿
- 慢事件订阅者不再直接拖慢主控制流程
- 离线连接多时，不再因同步 `Ping` 累积额外阻塞
- 网线拔掉但 socket 仍残留“已连接”状态时，连接能通过协议探活或首个失败 IO 被快速摘除
- 长时间执行中的 lease 不会被后台过期扫描误释放
- 现有 `ConnectionPool` 功能测试通过

### 第二阶段：补恢复吞吐，做真正的高并发控制面

#### 目标

- 将故障恢复从“全池扫描触发”改成“队列驱动”
- 支持几百离线连接并发恢复，同时把在线业务影响压低
- 将 `stop/restart/recover` 做成受控并发，而不是自然抢线程池

#### 改动项

1. 引入控制调度器

- 新增内部 `ConnectionControlScheduler`
- 基于 `Channel` 或等价轻量队列
- 控制命令至少覆盖：
  - `Recover`
  - `ForceClose`
  - `ForceReconnect`

2. 分离三类并发阈值

- `MaxConcurrentRecoveries`
- `MaxConcurrentForceCloses`
- `MaxConcurrentHealthChecks`

不要再让后台探活和强制控制共用一个并发阈值。

3. 故障恢复改队列驱动

- 条目进入 `Faulted` 后直接投递恢复任务
- 后台维护不再承担主恢复路径
- 后台维护只保留：
  - 租约过期回收
  - 空闲在线连接探活
  - 轻量清理

4. 为恢复退避增加抖动

- 保留冷却时间
- 改成指数退避或至少线性退避加随机抖动
- 避免几百条离线连接同拍重连

5. 控制面补批量接口

- `ForceReconnectManyAsync`
- `ForceCloseManyAsync`
- 视需要补 `RecoverManyAsync`

批量接口只负责提交命令和等待结果，不直接在调用线程串行跑每个条目。

6. 视情况限制单条目并发执行数

- 增加可选项：`MaxConcurrentExecutionsPerEntry`
- 默认建议为 `1`
- 目的不是降吞吐，而是避免同一设备的控制面和业务面在单连接上互相堆积

#### 第二阶段不做

- 不新增复杂分布式协调
- 不引入额外外部依赖
- 不做“自动最佳参数学习”

#### 第二阶段验收

- 数百离线连接恢复时，控制面响应时间稳定
- 在线设备读写延迟相对基线可控
- 故障恢复吞吐随配置的 worker 数可线性提升到合理范围
- 批量控制操作具备可观测性和结果汇总能力

## 推荐实施顺序

### 阶段一

1. 调整 `ResourcePool` 持锁范围
2. 事件异步化
3. 去掉同步 `Ping`
4. 补最小控制调度骨架

### 阶段二

1. 引入 `ConnectionControlScheduler`
2. 条目故障入恢复队列
3. 健康检查与恢复并发分离
4. 暴露批量异步控制接口
5. 增加压测和并发回归测试

## 测试计划

### 阶段一测试

- 慢事件订阅者不阻塞主流程
- 批量注销/释放/强制关闭时无明显全池卡顿
- 去除同步 `Ping` 后原有连接行为不回退
- 覆盖“网线拔掉但 socket 仍显示已连接”的半开连接场景
- 验证半开连接会被 `ProbeAsync` 或首个失败读写快速判定为 `Faulted`
- 覆盖长时间执行跨越 `LeaseTimeout` 的场景，验证执行中 lease 不会被回收

### 阶段二测试

- 100/300/500 离线连接并发恢复
- 在线连接持续读写，同时离线连接批量恢复
- 批量 `ForceClose` 与批量 `ForceReconnect` 压测
- 恢复退避抖动是否有效打散重连峰值

## 风险

- 事件异步化会改变订阅者观察到的精确时序
- 去掉同步 `Ping` 后，部分“网络通但协议端口未就绪”场景会更依赖协议探活和超时配置
- 半开连接识别如果仍过度依赖 `Connected`，会导致假在线条目滞留，必须在实现中明确收口
- 第二阶段引入调度器后，状态机边界必须补并发测试，不然容易出重复恢复、重复关闭、结果回写覆盖

## 完成定义

### 第一阶段完成

- 主流程去掉池级阻塞热点
- 事件分发不再同步拖慢业务/控制调用
- 代码结构为第二阶段队列化调度留出入口

### 第二阶段完成

- 故障恢复从扫描式变为队列式
- 控制面具备受控高并发能力
- 数百离线连接场景下，在线业务影响可测且可控
